NC中的事务

发表于2024-07-28|更新于2024-07-31
|阅读量:

NC中的事务管理

  • 在NCC开发中，通过.upm文件定义的 接口和实现类，能够开启事务。（tx = “CMT”开启事务，tx = “NONE”则不开启事务）

验证接口中的事务是够生效

  • 首先是最简单的一种，直接在后台任务中 获取实现类，执行插入方法。 （接口开启了事务 tx = “CMT”）
    • 不报错，正常插入
    • 报错，回滚

执行正常，没有报错的情况

报错，则回滚，数据库内没有插入数据

两个开启事务的接口，不会相互影响

  • 说明，两个接口按照顺序执行的情况，第二个接口报错是不会影响到第一个接口的事务的！除非这两个接口已经被包含在另一个事务之中了。

验证接口事务的独立性

接口中调用另外两个接口的插入方法

  • 两个接口都没有报错的情况，执行后~

其中，第二个接口执行后报错

  • 当第二个接口执行后报错，那么发生回滚，此时是否会影响到第一个接口中插入的数据 ？

第二个接口报错被内部catch住，没有抛出来的情况

如果不在内层，在外面一层去catch呢？

总结：由此，我们发现了一些值得注意的地方。

  1. 是否发生回滚的一个因素便是，开启了事务的接口的方法有没有抛出错误！

  1. 无论是多少层嵌套关系，一个接口调用另一个接口，再调用另一个接口，事务发生回滚的范围一定是第一个开启事务接口调用的方法的作用域。

  • 可以看到，从第一个接口调用开始，事务的范围就已经确定了，是第一个接口方法的作用域，接下来层层调用的接口方法，无论哪一层报错，都会整个回滚！（这里实际上后台任务就已经是开启了事务了，当时没有考虑到这一点~此时事务的作用范围是整个后台任务！）
  1. 即便是第三层，第十层，第100层接口（必须是接口抛出错误，普通方法抛出错误没用，除非连续抛出错误，抛到上一层接口去了。）内抛出了错误，都会被NC系统监察到，并视为此次事务中发生了报错，导致事务回滚，无论这个报错是否在上一层被捕获到了。

  • 那什么叫做普通方法抛出错误没用呢？（注意这里指普通方法抛错到上一层后，被catch住，并没有继续抛错）

  • 因为NC系统只能监视到开启事务管理的接口里面的方法报错，在上面的代码中，普通方法内报错后在上层捕获到，导致上层开启事务管理的接口，并没有抛出错误，所以NC系统视为当前事务正常执行，并没有发生回滚。

验证接口事务独立性_2

接口中调用一个接口，再调用一个普通方法

  • 执行正常，插入两条数据

普通方法中，抛出错误的情况

  • 结果会不会回滚呢？如何回滚的呢？

  • 实际上，这里普通方法抛出了错误，但是没有在上层被捕获处理掉，导致继续向上抛错误，这样就让开启了事务的接口方法抛出了错误，被NC事务系统发现了，导致的整个事务的回滚。

普通方法抛出错误，但是在上一层被捕获到！

NC中的独立事务 _RequiresNew

  • 在NC中，以接口实现类的方式（tx=“CMT”）是能够实现事务管理的，但是并不是独立事务！独立事务也是同样的定义方式，但是方法名字的后面要以 _RequiresNew 为结尾！

独立事务间是相互独立的

  • 事务与独立事务是相互隔离的，双方都不能读取到各自未提交的数据，但是当事务是嵌套的关系的时候，根据事务传播性，内层事务执行结束后不会自动提交，而是加入到外层事务之中，跟外层事务一起提交。（这里或许可以理解为，内层事务执行完成后会提交给外层事务？等提交到最外层以后再一起提交到数据库？）
  • 当内层的独立事务抛错时（在上一层捕获到，并没有继续抛出），则只会回滚独立事务，外层事务不受影响。
  • 当外层事务报错时，独立事务会单独提交，不会回滚。

提问：如果不是 事务中包含独立事务，只是事务中包含事务，则相互之间有隔离性吗？

事务中启用独立事务，则独立事务报错会如何？

  • 只要在上一次被catch捕获到，则不会影响外层的事务的执行，回滚只回滚独立事务的数据！

事务中启用独立事务，则事务报错会影响独立事务吗

  • 独立事务顺利执行结束，外层事务报错，则不会影响到独立事务中发生修改的数据！

事务代码分析

当前没有事务，开启一个新的事务时

  • 首先，当一个开启了事务管理的接口，通过 远程调用的 方式，执行一个方法时，假设当前并不处于一个事务之中，则执行这个方法会开启一个新的事务。
  • NC事务管理机制会接管这个方法，在执行方法之前，进入到 nc.bs.framework.ejb.CMTEJBServiceHandler
  • 上面的代码我们可以看到，会首先判断这个接口的方法名字是否以 _RequiresNew 结尾，我们这里调用的方法不是以这个结尾的，所以会执行下面的.delegate方法。
  • 上面我们关注 3 个比较重要的方法
    • beforeCallMethod(201); // 1. 开启事务前方法 *
    • o = _getBeanObject().delegate(arg0, arg1, arg2); // 2. 执行方法 *
    • afterCallMethod(201, (Exception)frameworkEJBException); // 3. 事务结束前方法 *

beforeCallMethod(201); //方法调用前

  • 上面有三个地方值得关注
    • 事务的传播类型是什么 ？
    • 事务的隔离级别是什么？
    • .begin（）方法中又做了什么？

getMethodTransectionType(methodId) //获取事务传播类型

  • 在这里，方法目前不处于事务之中，我们开启一个新的事务，且不以 _RequiresNew 结尾，此时我们拿到的事务传播类型是 1

  • 在下面的Begin方法中，会根据我们上边拿到的 int 值 转换成真正的 事务传播类型

  • SOURCE ：开启一个新的事务。如果一个事务已经存在，则先将这个存在的事务挂起 。

  • JOINED : 支持当前事务，如果当前没有事务，就新建一个事务。很好理解，不管有几个事务存在，都合并成一个事务来处理，只要有一个事务抛出异常，所有事务都会回滚；

注意： 在下面两种情况的时候，你创建的事务传播类型为 SOURCE

  1. 当前没有处于事务之中，而你创建了一个新的事务的时候
  2. 无论当前有没有处于事务之中，创建了一个独立事务

注意：如果，当前已经处于一个事务之中了，又开起了一个普通的事务（非独立事务），则此时事务传播类型为 JOINED

getMethodIsolateLevelType(methodId) //获取事务隔离级别

  • NC系统中，事务的隔离级别只有 2 这个类型，似乎整个NC系统都默认了 仅有的一种事务隔离级别。
  • 读已提交

这是大多数数据库系统的默认隔离级别（但不是MySQL默认的）。它满足了隔离的简单定义：一个事务只能看见已经提交事务所做的改变。这种隔离级别 也支持所谓的不可重复读（Nonrepeatable Read），因为同一事务的其他实例在该实例处理其间可能会有新的commit，所以同一select可能返回不同结果。

  • 如同上面讲过的，同样的select语句，在同一个事务之中，前后查询出来的结果不一致，因为有其他实例有新的commit。

.begin(this.currentMethodTransectionType, isolateLevel);// 开启事务

_getBeanObject().delegate(arg0, arg1, arg2); 动态代理执行目标方法

  • 这里就是以代理的方式，执行开启事务的目标方法了。

afterCallMethod(201, (Exception)frameworkEJBException); //方法调用后

  • 进入到 afterCallMethod方法
  • getIerpTransactionManagerProxy().end(exception);
